# Загрузка дня (day load из календаря)
> algo_version: n/a (ics_calendar.day_load, прозрачная формула) · источник данных: внешний API (ICS-фид календаря) · редактируемость: параметры в коде

## Что это

Балл 0-100 «насколько забит день» из личного календаря (ICS-подписка, без OAuth). Контекст-сигнал того же класса, что погода или поездки: помогает объяснить просадку recovery, ничего не диагностирует.

## Формула / алгоритм

```
day_load_score = 70 × min(busy_hours / 8, 1)        — часы встреч против 8-часового рабочего дня
              + 20 × min(meetings_count / 8, 1)      — переключения контекста
              + 10, если встреч >= 3 и нет ни одного окна >= 1 ч  — нет «окна восстановления»
```

Кап на 100. Разложение возвращается в `score_parts` (busy_hours / meetings / no_recovery_gap) — балл объясним, не чёрный ящик.

Детали подсчёта:

- пересекающиеся встречи **сливаются** перед подсчётом busy-часов — двойная бронь не учитывается дважды;
- встречи обрезаются границами локального дня;
- all-day события busy-часов не добавляют (перечисляются отдельно);
- «окно восстановления» — разрыв >= 60 мин между слитыми занятыми интервалами.

Парсинг ICS — RFC 5545 MVP: VEVENT, all-day, таймзоны (Z/TZID/floating), RRULE только DAILY/WEEKLY в окне ±7 дней, EXDATE, RECURRENCE-ID; всё непокрытое честно попадает в warnings.

## Параметры (константы кода)

| параметр | значение | где в коде | зачем |
|---|---|---|---|
| вес busy-часов | 70 | `openhealth/connectors/ics_calendar.py: day_load` | главный фактор: сколько дня съели встречи |
| нормализатор часов | 8.0 | `openhealth/connectors/ics_calendar.py: WORKDAY_HOURS` | 8 ч встреч = полные 70 пунктов |
| вес числа встреч | 20 | `openhealth/connectors/ics_calendar.py: day_load` | цена переключений контекста |
| нормализатор встреч | 8 | `openhealth/connectors/ics_calendar.py: MEETINGS_NORM` | 8 встреч = полные 20 пунктов |
| штраф фрагментации | 10 | `openhealth/connectors/ics_calendar.py: day_load` | >= 3 встреч без часового окна = день без передышки |
| минимальное окно | 60 мин | `openhealth/connectors/ics_calendar.py: GAP_MIN_MINUTES` | меньше часа — не восстановление |
| окно RRULE | ±7 дней | `openhealth/connectors/ics_calendar.py: RRULE_WINDOW_DAYS` | повторы разворачиваются только вокруг «сейчас» |

## Источники и доверие

- Формула — наша эвристика, прозрачная и тюнябельная; никакой претензии на научность весов 70/20/10, это осознанная калибровка «что делает день тяжёлым».
- ICS URL — секрет (даёт чтение календаря): хранится только в `~/.openhealth/calendar.json` (0600), никогда не логируется и не попадает в ошибки.

## Известные ограничения

- Календарь видит встречи, а не реальную нагрузку: пустой календарь в день дедлайна покажет 0.
- RRULE кроме DAILY/WEEKLY не разворачивается (помечается `recurring_skipped`).
- Балл — контекст для чтения recovery, не самостоятельная метрика здоровья.
