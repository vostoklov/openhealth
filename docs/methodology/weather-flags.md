# Погодные флаги
> algo_version: n/a (connectors/weather, пороги-константы) · источник данных: внешний API (Open-Meteo) + движок · редактируемость: параметры в коде

## Что это

Осторожные, грейдированные флаги погоды как ко-фактора восстановления: «давление падает 9 гПа — возможный плохой день для метеочувствительных». Погода — контекст того же класса, что календарь и поездки; никогда не диагноз. Данные — Open-Meteo (бесплатно, без ключа), домашняя локация в `~/.openhealth/weather.json` (0600).

## Формула / алгоритм

По каноническому дню (t_min/t_max/t_mean, среднее давление и его суточное изменение, влажность, осадки, ветер) выставляются флаги по порогам. Каждый флаг несёт грейд доверия и русское сообщение, явно проговаривающее неуверенность:

| флаг | условие | грейд (популяционный) |
|---|---|---|
| pressure_drop | изменение давления за 24 ч <= −8 гПа | C3 (наблюдательные данные смешанные) |
| heat | t_max >= 30° | **C4** — жара ухудшает сон, устойчивый результат (самое сильное наше утверждение) |
| cold | t_min <= 0° | C2 — просто контекст, влияние не доказано |
| humidity | средняя влажность >= 85% | C2 — значимо в основном в связке с теплом |
| precipitation | осадки >= 1 мм | C2 — могла выпасть привычная прогулка |

**Персональные грейды.** Если у пользователя в профиле чувствительность к флагу: `declared` (заявлена) → личный паттерн, cap C2; `validated` (повторялась on/off) → может подняться до C3. Без записи — популяционный грейд из таблицы.

**Мост к корреляциям**: каждый фактор превращается в булеву «привычку» (день с падением давления да/нет, жаркий день да/нет...) и идёт в тот же движок корреляций с порогом 5/5 — «падает ли мой recovery в дни падения давления» считается на личных данных (см. correlations.md).

`pressure_change_24h` = среднее давление дня минус среднее давление вчерашнего дня (средние — по часовым значениям); для первого дня диапазона запрашивается день-довесок.

## Параметры (константы кода)

| параметр | значение | где в коде | зачем |
|---|---|---|---|
| падение давления | 8.0 гПа/24ч | `openhealth/connectors/weather.py: PRESSURE_DROP_HPA` | >= 8 гПа/сутки — «быстрое падение» в синоптике (сильный фронт) |
| жара | 30.0 °C | `openhealth/connectors/weather.py: HEAT_T_MAX_C` | дневная жара >= 30° деградирует ночной сон (исследования сна) |
| холод | 0.0 °C | `openhealth/connectors/weather.py: COLD_T_MIN_C` | минус — просто контекст дня |
| влажность | 85.0 % | `openhealth/connectors/weather.py: HUMIDITY_HIGH_PCT` | сама по себе слабый сигнал |
| осадки | 1.0 мм | `openhealth/connectors/weather.py: RAIN_MM` | достаточно, чтобы сорвать прогулку |

## Источники и доверие

- Жара → плохой сон: консистентные исследования сна — C4 («вероятно»).
- Давление → головная боль/суставы: наблюдательные и смешанные популяционные данные — максимум C3; личный сырой паттерн — C2.
- Каждое сообщение в UI прямо называет грейд и неуверенность; правило кап-а — `openhealth/evidence.py`.

## Известные ограничения

- Погода по домашней локации: поездки в другой климат флаги не догоняют (см. travel-контекст).
- Open-Meteo: forecast-эндпоинт покрывает ~92 дня назад, дальше — архив ERA5 (переключение зашито, но модели данных слегка разные).
- Метеочувствительность научно зыбкая — флаги намеренно сформулированы как гипотезы для проверки на своих данных, не как факты.
